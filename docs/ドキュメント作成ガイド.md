# 小規模プロジェクトのドキュメント構成ガイド

## 推奨ドキュメント構成

小規模プロジェクト（個人開発、小規模チーム）では、**必要最小限のドキュメント**に絞ることで、メンテナンス性と実用性を両立できます。

## 必須ドキュメント（3つ）

### 1. README.md（プロジェクトルート）

**目的**: プロジェクトの概要、セットアップ、基本的な使い方

**内容**:
- プロジェクトの説明
- 技術スタック
- セットアップ手順
- 実行方法
- 基本的な使用例
- 必要な環境

**理由**: 最初に見るドキュメント。プロジェクトの全体像を把握できる。

### 2. 設計書（docs/基本設計書.md）

**目的**: アーキテクチャと設計方針の説明

**内容**:
- アーキテクチャの概要（クリーンアーキテクチャなど）
- レイヤー構成と責務
- プロジェクト構造
- 主要な設計判断
- API仕様（WebAPIの場合）

**理由**: コードを理解するための設計思想を文書化。新規メンバーのオンボーディングに必要。

**補足**: 詳細な実装設計が必要な場合は、`docs/詳細設計書.md`も参照。

### 3. API仕様（docs/API.md または Swagger）

**目的**: APIエンドポイントの詳細仕様

**内容**:
- エンドポイント一覧
- リクエスト/レスポンス形式
- エラーコード
- 使用例

**理由**: WebAPIの場合、外部から利用する際の仕様書として必要。

**補足**: Swagger/OpenAPIを使用している場合は、`/swagger`で自動生成されるため、別途ドキュメントは不要な場合もあり。

## オプショナルドキュメント

### 開発者向け

- **CHANGELOG.md**: バージョンごとの変更履歴（リリースする場合）
- **CONTRIBUTING.md**: コントリビューションガイド（オープンソースの場合）
- **docs/詳細設計書.md**: 詳細なクラス設計と実装詳細

### 運用向け

- **docs/DEPLOYMENT.md**: デプロイ手順（本番環境にデプロイする場合）
- **docs/TROUBLESHOOTING.md**: トラブルシューティング（よくある問題と解決方法）

### Git関連（初心者向け）

- **docs/Gitセットアップガイド.md**: Gitの基本的な使い方（チーム開発の場合）

## 不要なドキュメント（小規模プロジェクト）

以下のドキュメントは、小規模プロジェクトでは**過剰**です：

- ❌ 詳細なクラス設計書（コードが自己文書化されている）
- ❌ 詳細なテスト計画書（テストコードで十分）
- ❌ 詳細な運用マニュアル（シンプルな運用なら不要）
- ❌ 詳細なセキュリティポリシー（小規模ならREADMEに簡潔に記載）
- ❌ 詳細なパフォーマンステスト結果（必要になったら作成）

```
img-resizer/
├── README.md                    # 必須：プロジェクト概要、セットアップ
├── docs/
│   ├── 基本設計書.md             # 必須：アーキテクチャ設計
│   ├── 詳細設計書.md             # 推奨：詳細な実装設計

└── （その他のドキュメントは必要に応じて）
```

## ドキュメントのメンテナンス

### 原則

1. **コードと同期**: コードが変更されたら、関連ドキュメントも更新
2. **シンプルに**: 複雑な説明より、シンプルで明確な説明を
3. **例を重視**: 説明より、実際の使用例を優先
4. **重複を避ける**: 同じ情報を複数の場所に書かない

### 更新タイミング

- **README.md**: プロジェクトの主要な変更時
- **設計書**: アーキテクチャが変更された時
- **API仕様**: エンドポイントが追加・変更された時

## まとめ

小規模プロジェクトでは、**README.md**と**設計書**の2つがあれば十分です。WebAPIの場合は**API仕様**も追加すると良いでしょう。

過剰なドキュメントはメンテナンス負担になるため、必要最小限に留めることが重要です。
